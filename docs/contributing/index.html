Contributing

Contributing to FinOps-as-Code

This guide sets the contribution standard expected across FinOps-as-Code projects: useful automation, readable code, testing, safe defaults, and a real connection to cost, efficiency, and governance outcomes.

English Markdown Portuguese Markdown Portuguese HTML
FinOps is not only visibility. It is also execution. Contributions should help teams act on cost, efficiency, and governance opportunities in practical ways.

Project Vision

FinOps-as-Code currently prioritizes simple, executable, and independent scripts. That focus is intentional: this format makes adoption, review, reuse, and validation of value much easier across different environments.

Contribution Principles

  • Solve a real FinOps problem related to cost reduction, efficiency, governance, or operational visibility.
  • Prefer simple and direct implementations over unnecessary abstraction.
  • Keep contributions understandable, reusable, maintainable, and safe to review.
  • Use configuration and parameters instead of hardcoded environment-specific values.
  • Include clear logging or output so users can understand what the automation is doing.
  • Write for an open source audience, not only for the original author or team.

Types of Contributions

  • Scripts: the main contribution type today, always tied to a real FinOps problem.
  • Documentation: to improve clarity, adoption, maintainability, and onboarding.
  • Bug fixes: with clear explanation of expected versus actual behavior.

Engineering Expectations

  • All submitted code must be tested.
  • Code must follow software engineering standards, coding standards, and good maintenance practices.
  • Scripts should support safe defaults and dry-run behavior when meaningful.
  • Contributions must avoid sensitive data, private endpoints, secrets, tokens, and account-specific details.

Best Practices for Scripts

  • Prefer one script solving one clear problem.
  • Use parameters or configuration instead of hardcoded values.
  • Avoid destructive actions unless they are explicitly enabled.
  • Include enough logging for understanding and operation.
  • Document prerequisites, inputs, and expected behavior.

Pull Request Guidance

Every pull request should clearly explain:

  • what the contribution does
  • what FinOps problem it solves
  • how it was tested
  • what assumptions, limitations, or safety considerations exist
  • what documentation needs to be updated alongside it

Open Source Support and Responsibility

Code in FinOps-as-Code repositories is provided on an as is basis. Maintainers and contributors do not guarantee support, SLA, or ongoing maintenance. Adopters are responsible for validating behavior, compatibility, safety, and operational impact in their own environments.

Closing Philosophy

The strongest contributions are the ones that turn an important financial operations problem into something practical, understandable, and reusable. If the work helps teams act better on FinOps priorities, it is probably moving in the right direction.
