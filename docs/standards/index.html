Standards

Shared Standards for Practical FinOps Automation

Lightweight engineering expectations for code that is reusable, maintainable, understandable, and safe to run in different environments.

What this page emphasizes

Clarity Contributions should be easy to inspect, understand, and operate.
Safety Safe defaults, dry-run, and predictable behavior are expected.
Reuse Code should be portable beyond the original environment where it was written.

Core expectations

  • Prefer simple, focused contributions over heavy abstractions.
  • Favor standalone scripts that solve one clear FinOps problem.
  • Use parameters and configuration instead of hardcoded environment details.
  • Require testing for submitted code.
  • Keep code readable, maintainable, and ready for reuse in an open context.

The goal is not to make contribution heavy. The goal is to preserve a minimum engineering bar so other teams can inspect, adapt, and run the automation with confidence.

Safety expectations

  • Default to safe behavior.
  • Use dry-run when meaningful.
  • Avoid destructive actions unless explicitly enabled.
  • Provide logging or clear output for operator awareness.
  • Make error handling understandable and predictable.

Wherever possible, scripts should fail in explicit and understandable ways instead of taking irreversible action without enough context for the operator.

Sensitive data handling

Repositories should not include:

  • account IDs
  • tokens or secrets
  • private endpoints
  • internal-only configuration
  • organization-specific confidential information

When examples are needed, use sanitized placeholders and fictional values. The repository should stay publishable and reusable without exposing private details from the original environment.

Fit for open reuse

Contributions should be written for reuse outside the original environment. That means documenting assumptions, avoiding private context, and making scripts easy for others to inspect and adapt.

What this means in practice

  • use clear names for parameters, flags, and outputs
  • document prerequisites and execution limits
  • avoid hidden dependencies or tight coupling to the author’s environment
  • keep dependencies to the minimum necessary
  • prefer one script solving one clear problem over broad flows that are hard to review

Checklist

Dry-run Include it whenever destructive or high-impact actions are possible.
Logging Make it clear what the script is doing and what it decided not to do.
Parameters Prefer explicit flags and configuration over hardcoded values.
