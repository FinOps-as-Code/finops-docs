Standards

Shared Standards for Practical FinOps Automation

Lightweight engineering expectations for code that is reusable, maintainable, understandable, and safe to run in different environments.

Portuguese version

Core expectations

  • Prefer simple, focused contributions over heavy abstractions.
  • Favor standalone scripts that solve one clear FinOps problem.
  • Use parameters and configuration instead of hardcoded environment details.
  • Require testing for submitted code.
  • Keep code readable, maintainable, and ready for reuse in an open context.

Safety expectations

  • Default to safe behavior.
  • Use dry-run when meaningful.
  • Avoid destructive actions unless explicitly enabled.
  • Provide logging or clear output for operator awareness.
  • Make error handling understandable and predictable.

Sensitive data handling

Repositories should not include:

  • account IDs
  • tokens or secrets
  • private endpoints
  • internal-only configuration
  • organization-specific confidential information

Fit for open reuse

Contributions should be written for reuse outside the original environment. That means documenting assumptions, avoiding private context, and making scripts easy for others to inspect and adapt.
