Overview

FinOps-as-Code Project Vision

FinOps-as-Code exists to turn FinOps decisions into practical automation. The initiative starts from a simple point: visibility alone does not remove waste. Teams need simple, executable, and safe code they can use to act on cost, efficiency, and governance opportunities.

What this page covers

This page explains the current focus of the initiative and the kind of work that fits well here.

Current focus Safe, practical automation for real cloud financial operations problems.
Project fit Code that is useful, understandable, and realistic for others to adapt.

Current focus

The current direction is to publish automation that solves real cloud financial operations problems with clear execution paths. Today that appears mainly as scripts and lean workflows, because that format is the fastest way to prove usefulness, adoption, and reuse.

That focus is intentional. Scripts are often the most accessible way to validate ideas quickly, support real operating teams, and keep contributions understandable across different environments and maturity levels.

What the initiative publishes today

  • cleanup_block_storage_idle targets unused block storage resources that continue to generate avoidable cloud cost.
  • cleanup_ip_idle focuses on reclaiming idle IP-related resources that create waste or weaken governance.
  • linktree-f2-brazil supports the public visibility layer of the organization.

What fits the initiative

A strong project for FinOps-as-Code should solve a concrete cost, efficiency, or governance problem, be easy to understand, safe to evaluate, and realistic for other teams to adapt. The goal is not to limit work to the current repositories. The goal is to keep a clear bar for what is worth publishing under this initiative.

That means preferring practical value over framework complexity and writing code that other teams can inspect, test, and run without inheriting hidden assumptions from the original author’s environment.

How the initiative can grow

New repositories can be created whenever a legitimate FinOps use case deserves its own automation. What should stay consistent is the contribution bar itself: practical implementation, reusable engineering, safe defaults, and a clear connection to real FinOps outcomes.
